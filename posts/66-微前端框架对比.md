## 微前端简介

微前端（Micro-Frontends）是一种将前端应用拆分为多个独立、可部署的子应用的架构理念，借鉴了后端微服务的思想。

### 核心概念

```
                    ┌─────────────────────────────────┐
                    │         主应用（Shell App）        │
                    │  ┌──────────┐  ┌──────────────┐  │
                    │  │  子应用A  │  │   子应用B     │  │
                    │  │  (React) │  │   (Vue)      │  │
                    │  └──────────┘  └──────────────┘  │
                    │  ┌──────────────────────────────┐ │
                    │  │         子应用C (Angular)      │ │
                    │  └──────────────────────────────┘ │
                    └─────────────────────────────────┘
```

### 微前端的六大特征

| 特征 | 说明 |
|------|------|
| **技术栈无关** | 各子应用可使用不同框架（React/Vue/Angular） |
| **独立开发** | 各团队可独立开发，互不干扰 |
| **独立部署** | 各子应用可单独构建、发布 |
| **独立运行时** | 子应用间状态隔离，互不影响 |
| **增量升级** | 支持老旧系统逐步迁移 |
| **统一体验** | 用户感知上仍是一个完整应用 |

## 为什么需要微前端

### 传统巨石应用的痛点

```javascript
// 典型巨石应用问题
// 1. 单仓库代码量巨大，编译慢
// 2. 技术栈升级困难，牵一发动全身
// 3. 团队协作冲突频繁
// 4. 部署风险高，一个 bug 影响全局
// 5. 难以局部扩展或替换
```

### 微前端的收益

```
巨石应用                    微前端架构
────────────────           ──────────────────────
单一代码仓库         →     多个独立仓库
统一技术栈           →     技术栈自由选择
整体发布             →     独立发布
全局影响             →     隔离影响范围
团队耦合             →     团队自治
```

## 主流微前端框架

### 1. single-spa

**GitHub**: https://github.com/single-spa/single-spa  
**Stars**: 12k+  
**定位**: 微前端领域的先驱，提供基础路由调度能力

```javascript
// single-spa 注册子应用示例
import { registerApplication, start } from 'single-spa'

// 注册 Vue 子应用
registerApplication({
  name: 'vue-app',
  app: () => import('./vue-app/main.js'),
  activeWhen: (location) => location.pathname.startsWith('/vue'),
  customProps: {
    domElement: document.getElementById('vue-container'),
    sharedStore: store
  }
})

// 注册 React 子应用
registerApplication({
  name: 'react-app',
  app: () => import('./react-app/main.js'),
  activeWhen: (location) => location.pathname.startsWith('/react')
})

start({
  urlRerouteOnly: true // 只在 URL 变化时重新路由
})
```

**优点**：
- 最早的微前端框架，社区成熟
- 支持多种框架（React/Vue/Angular）
- 路由驱动，接入成本较低

**缺点**：
- 样式隔离能力弱
- JS 沙箱需自行实现
- 子应用改造成本较高

### 2. qiankun（乾坤）

**GitHub**: https://github.com/umijs/qiankun  
**Stars**: 15k+  
**定位**: 基于 single-spa 封装，蚂蚁集团出品，国内使用最广泛

```javascript
// qiankun 主应用配置
import { registerMicroApps, start, initGlobalState } from 'qiankun'

// 初始化全局状态
const { onGlobalStateChange, setGlobalState } = initGlobalState({
  user: null,
  token: ''
})

// 注册子应用
registerMicroApps([
  {
    name: 'react-app',        // 子应用名称
    entry: '//localhost:7100', // 子应用入口
    container: '#subapp-viewport', // 挂载容器
    activeRule: '/react',     // 激活路由
    props: {                  // 传递给子应用的数据
      routerBase: '/react',
      getGlobalState: () => store.getState()
    }
  },
  {
    name: 'vue-app',
    entry: '//localhost:7200',
    container: '#subapp-viewport',
    activeRule: '/vue'
  }
], {
  // 生命周期钩子
  beforeLoad: [(app) => console.log('before load', app.name)],
  beforeMount: [(app) => console.log('before mount', app.name)],
  afterMount: [(app) => console.log('after mount', app.name)],
  beforeUnmount: [(app) => console.log('before unmount', app.name)],
  afterUnmount: [(app) => console.log('after unmount', app.name)]
})

start({
  sandbox: {
    strictStyleIsolation: true, // 严格样式隔离（Shadow DOM）
    experimentalStyleIsolation: true // 实验性样式隔离（scoped css）
  },
  prefetch: 'all' // 预加载所有子应用
})
```

```javascript
// qiankun 子应用（Vue 3）改造
import { createApp } from 'vue'
import App from './App.vue'
import router from './router'

let app = null

// 独立运行时
if (!window.__POWERED_BY_QIANKUN__) {
  createApp(App).use(router).mount('#app')
}

// 微前端生命周期
export async function bootstrap() {
  console.log('vue app bootstraped')
}

export async function mount(props) {
  const { container, routerBase } = props
  
  app = createApp(App)
  app.use(router)
  
  // 修改路由 base
  router.options.history = createWebHistory(routerBase || '/')
  
  app.mount(container ? container.querySelector('#app') : '#app')
}

export async function unmount() {
  app?.unmount()
  app = null
}
```

**优点**：
- 开箱即用，配置简单
- 完善的 JS 沙箱（Proxy + Snapshot）
- 样式隔离方案成熟
- 国内文档和社区活跃

**缺点**：
- 基于 HTML Entry，有一定性能开销
- 不支持 Vite 子应用（需额外插件）
- 沙箱有一定局限性

### 3. micro-app（京东）

**GitHub**: https://github.com/micro-zoe/micro-app  
**Stars**: 5k+  
**定位**: 基于 Web Components，京东出品，对 Vite 友好

```javascript
// micro-app 主应用 - 极简使用
import microApp from '@micro-zoe/micro-app'
microApp.start()

// 在模板中使用 Web Component 标签即可
// <micro-app name='vue-app' url='http://localhost:3001/'></micro-app>
```

```vue
<!-- 主应用模板 -->
<template>
  <div>
    <h1>主应用</h1>
    <!-- 像使用原生标签一样使用子应用 -->
    <micro-app
      name="vue-app"
      url="http://localhost:3001/"
      :data="{ token: userToken }"
      @datachange="handleDataChange"
      baseroute="/vue-app"
    ></micro-app>
  </div>
</template>

<script setup>
import { ref } from 'vue'
const userToken = ref('xxx-token')

const handleDataChange = (e) => {
  console.log('子应用数据变化:', e.detail.data)
}
</script>
```

```javascript
// micro-app 子应用通信
// 子应用向主应用发送数据
window.microApp?.dispatch({ type: 'USER_LOGIN', payload: { userId: 1 } })

// 主应用向子应用发送数据（通过 data 属性）
// 子应用监听
window.microApp?.addDataListener((data) => {
  console.log('收到主应用数据:', data)
})
```

**优点**：
- 基于 Web Components，无侵入接入
- 对 Vite 原生支持
- 接入成本极低，子应用改造少
- 虚拟路由系统

**缺点**：
- 相对较新，生产案例较少
- 部分 API 仍在迭代

### 4. Module Federation（模块联邦）

**定位**: Webpack 5 内置功能，从打包层面实现微前端

```javascript
// webpack.config.js - 主应用（Host）配置
const { ModuleFederationPlugin } = require('webpack').container

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'host',
      remotes: {
        // 声明远程模块
        'remote-app': 'remoteApp@http://localhost:3001/remoteEntry.js',
        'another-app': 'anotherApp@http://localhost:3002/remoteEntry.js'
      },
      shared: {
        // 共享依赖，避免重复加载
        react: { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom': { singleton: true, requiredVersion: '^18.0.0' }
      }
    })
  ]
}

// 主应用中使用远程组件
import React, { lazy, Suspense } from 'react'

// 动态导入远程模块
const RemoteButton = lazy(() => import('remote-app/Button'))
const RemotePage = lazy(() => import('another-app/Page'))

function App() {
  return (
    <Suspense fallback="Loading...">
      <RemoteButton text="远程按钮" />
      <RemotePage />
    </Suspense>
  )
}
```

```javascript
// webpack.config.js - 子应用（Remote）配置
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'remoteApp',
      filename: 'remoteEntry.js', // 入口文件
      exposes: {
        // 暴露的模块
        './Button': './src/components/Button',
        './Page': './src/pages/HomePage'
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true }
      }
    })
  ]
}
```

```javascript
// Vite 中使用 Module Federation（@originjs/vite-plugin-federation）
// vite.config.ts - 主应用
import federation from '@originjs/vite-plugin-federation'

export default {
  plugins: [
    federation({
      name: 'host-app',
      remotes: {
        'remote-app': 'http://localhost:5001/assets/remoteEntry.js'
      },
      shared: ['vue', 'vue-router']
    })
  ]
}
```

**优点**：
- Webpack 5 原生支持，无需额外框架
- 代码级别共享，性能最优
- 支持运行时动态加载
- 依赖共享优化，减少重复加载

**缺点**：
- 强依赖 Webpack（Vite 支持需插件）
- 样式和 JS 隔离需自行处理
- 调试复杂度较高

### 5. Garfish（字节跳动）

**GitHub**: https://github.com/modern-js-dev/garfish  
**Stars**: 2k+  
**定位**: 字节跳动内部微前端框架开源版

```javascript
import Garfish from 'garfish'

Garfish.run({
  domGetter: '#subapp',
  basename: '/',
  apps: [
    {
      name: 'vue-app',
      activeWhen: '/vue',
      entry: 'http://localhost:3001',
      props: { msg: 'hello' }
    }
  ],
  sandbox: {
    open: true,        // 开启沙箱
    snapshot: false,   // 快照沙箱（false = Proxy 沙箱）
    fixBaseUrl: true   // 修复资源路径
  }
})
```

## 框架深度对比

### 综合对比表

| 维度 | single-spa | qiankun | micro-app | Module Federation | Garfish |
|------|-----------|---------|-----------|-------------------|---------|
| **出品方** | 社区 | 蚂蚁 | 京东 | Webpack/Meta | 字节 |
| **实现原理** | JS Entry | HTML Entry | Web Components | Webpack 模块 | HTML Entry |
| **JS 沙箱** | ❌ 需自行 | ✅ Proxy+Snapshot | ✅ | ❌ | ✅ |
| **样式隔离** | ❌ | ✅ Shadow DOM/Scoped | ✅ Shadow DOM | ❌ | ✅ |
| **Vite 支持** | ⚠️ 需配置 | ⚠️ 需插件 | ✅ 原生 | ✅ 需插件 | ⚠️ |
| **接入成本** | 中 | 中 | 低 | 低 | 中 |
| **通信机制** | props/事件 | GlobalState/props | data/事件 | 直接引用 | Channel |
| **社区活跃度** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **国内案例** | 一般 | 最多 | 较多 | 较多 | 较多 |

### 隔离方案对比

```
JS 隔离方案演进：
─────────────────────────────────────────────────
1. 快照沙箱（Snapshot Sandbox）
   原理：激活时保存 window 快照，卸载时恢复
   缺点：不支持并发运行多个子应用

2. Proxy 沙箱（Proxy Sandbox）
   原理：为每个子应用创建 Proxy 代理 window
   优点：支持并发，隔离彻底
   qiankun / micro-app 均使用此方案

3. With 沙箱
   原理：通过 with(fakeWindow) 包裹代码执行
   (function(window, self, globalThis) { /* 子应用代码 */ })(fakeWindow)
─────────────────────────────────────────────────

样式隔离方案：
─────────────────────────────────────────────────
1. Shadow DOM（严格隔离）
   - 浏览器原生支持
   - 完全隔离，但与全局样式不通
   
2. Scoped CSS（实验性）
   - 动态添加 data-qiankun 属性选择器前缀
   - 兼容性好，但有一定 hack 成本

3. CSS Modules / CSS-in-JS
   - 从源码层面解决冲突
─────────────────────────────────────────────────
```

### 通信方案对比

```javascript
// 1. qiankun - GlobalState（发布订阅）
// 主应用
import { initGlobalState } from 'qiankun'
const actions = initGlobalState({ count: 0 })
actions.onGlobalStateChange((state, prev) => {
  console.log('主应用感知变化:', state, prev)
})
actions.setGlobalState({ count: 1 })

// 子应用
export async function mount(props) {
  const { onGlobalStateChange, setGlobalState } = props
  onGlobalStateChange((state) => {
    console.log('子应用感知变化:', state)
  })
}

// 2. micro-app - 事件通信
// 主应用 → 子应用（data 属性）
// <micro-app :data="{ user: currentUser }"></micro-app>

// 子应用 → 主应用
window.microApp.dispatch({ type: 'LOGOUT' })

// 主应用监听
document.querySelector('micro-app').addEventListener('datachange', (e) => {
  handleChildMessage(e.detail.data)
})

// 3. Module Federation - 直接共享 Store
// 共享 Redux/Pinia Store，最彻底的通信方式
import sharedStore from 'remote-app/store'
sharedStore.dispatch({ type: 'SET_USER', payload: user })
```

## 适用场景分析

### qiankun 适合

```
✅ 大型企业中后台系统改造
✅ 多团队协作，各团队技术栈不同
✅ 存量 jQuery/Angular 老系统渐进迁移
✅ 需要成熟稳定方案，团队没有微前端经验
✅ 主应用为 Webpack 构建
```

### micro-app 适合

```
✅ 新项目，希望接入成本最低
✅ 子应用是 Vite 构建
✅ 追求更现代的 Web Components 方案
✅ 子应用改造资源受限
```

### Module Federation 适合

```
✅ 需要模块级别共享（共享组件/工具函数）
✅ 技术栈统一（同为 React 或 Vue）
✅ 对性能要求高，需要避免重复加载
✅ 团队对 Webpack 生态熟悉
```

### single-spa 适合

```
✅ 需要高度定制化的微前端方案
✅ 团队有足够的架构能力
✅ 在 single-spa 生态基础上扩展
```

## 选型建议

### 决策流程图

```
开始选型
   │
   ├─ 子应用是否使用 Vite 构建？
   │    ├─ 是 → micro-app 或 Module Federation（+vite-plugin-federation）
   │    └─ 否 ↓
   │
   ├─ 是否有 JS/CSS 隔离需求？
   │    ├─ 有 → qiankun 或 micro-app
   │    └─ 无 → Module Federation（纯模块共享场景）
   │
   ├─ 团队规模和经验？
   │    ├─ 国内团队，有微前端经验 → qiankun（生态最完善）
   │    ├─ 追求简单接入 → micro-app
   │    └─ 高度定制需求 → single-spa
   │
   └─ 结果
```

### 推荐组合

```
场景1：大型企业后台（推荐 qiankun）
  主应用: Vue 3 + Vite（做主框架）
  子应用A: React（独立团队）
  子应用B: Vue 2（老系统迁移）
  子应用C: Angular（历史遗留）

场景2：中小型新项目（推荐 micro-app）
  所有应用使用 Vue 3 + Vite
  接入成本最低，效果好

场景3：组件库共享（推荐 Module Federation）
  各应用共享设计系统组件
  共享 React / Vue 依赖
  减少包体积
```

### 技术选型 Checklist

- [ ] 评估现有系统技术栈（Webpack / Vite）
- [ ] 确认隔离需求级别（JS 隔离 / 样式隔离）
- [ ] 评估子应用改造成本
- [ ] 考察团队微前端经验
- [ ] 调研框架社区活跃度和维护状态
- [ ] 制作 POC 验证方案可行性
- [ ] 明确子应用通信需求

## 总结

| 框架 | 推荐指数 | 最适合场景 |
|------|---------|-----------|
| **qiankun** | ⭐⭐⭐⭐⭐ | 国内企业级项目首选，生态最完善 |
| **micro-app** | ⭐⭐⭐⭐ | 新项目低成本接入，Vite 友好 |
| **Module Federation** | ⭐⭐⭐⭐ | 模块共享、同技术栈项目 |
| **single-spa** | ⭐⭐⭐ | 高度定制场景 |
| **Garfish** | ⭐⭐⭐ | 字节系技术栈 |

> **2024 年推荐**：国内项目首选 **qiankun**，追求简单接入选 **micro-app**，纯模块共享选 **Module Federation**。
