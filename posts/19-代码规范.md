## 一、代码规范

### 1. 命名规范

#### 变量和函数命名
```javascript
// ✅ 正确：驼峰命名法
const userName = 'John'
const isLoggedIn = true

function getUserById(id) {
  // ...
}

// ❌ 错误：下划线命名（除非特殊约定）
const user_name = 'John'  // 不推荐

// ✅ 正确：布尔值使用 is/has/can 前缀
const isActive = true
const hasPermission = false
const canEdit = true

// ❌ 错误：布尔值没有前缀
const active = true  // 不推荐

// ✅ 正确：常量使用大写下划线
const API_BASE_URL = 'https://api.example.com'
const MAX_RETRY_COUNT = 3

// ❌ 错误：常量使用驼峰
const apiBaseUrl = 'https://api.example.com'  // 不推荐
```

**解释**：
- 变量和函数使用驼峰命名法（camelCase）
- 布尔值使用 `is`/`has`/`can` 前缀，提高可读性
- 常量使用大写下划线命名（UPPER_SNAKE_CASE），表示不可变

---

#### 类名命名
```javascript
// ✅ 正确：帕斯卡命名法（首字母大写）
class UserService {
  // ...
}

class User {
  // ...
}

// ❌ 错误：类名使用驼峰
class userService {  // 不推荐
  // ...
}
```

**解释**：类名使用帕斯卡命名法（PascalCase），与变量和函数区分。

---

#### 私有属性命名
```javascript
// ✅ 正确：使用 # 前缀（ES2022）
class User {
  #privateField = 'private'

  #privateMethod() {
    // ...
  }
}

// ✅ 或使用 _ 前缀（约定俗成）
class User {
  _privateField = 'private'

  _privateMethod() {
    // ...
  }
}

// 解释：# 是真正的私有属性，_ 是约定俗成的私有属性
```

**解释**：
- `#` 是 ES2022 的私有属性语法，真正私有，外部无法访问
- `_` 是约定俗成的私有属性，不是真正的私有，只是提醒开发者

---

#### 文件命名
```javascript
// ✅ 正确：kebab-case（短横线命名）
// user-service.js
// api-client.js
// user-profile.vue

// ✅ 或 camelCase（React 组件常用）
// UserService.js
// UserProfile.jsx

// ❌ 错误：混合命名
// user_service.js  // 不推荐
// userProfile.vue  // 如果约定使用 kebab-case
```

**解释**：
- JavaScript 文件通常使用 kebab-case（短横线）
- React 组件文件通常使用 PascalCase
- 保持项目命名风格一致

### 2. 注释规范

#### 单行注释
```javascript
// ✅ 正确：解释为什么，而不是是什么
// 使用正则表达式验证邮箱格式，因为需要更严格的验证规则
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/

// ❌ 错误：解释显而易见的代码
// 定义邮箱正则
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/
```

**解释**：注释应该解释"为什么"，而不是"是什么"。代码本身已经说明了"是什么"。

---

#### 多行注释
```javascript
/**
 * 计算两个数的和
 *
 * @param {number} a - 第一个数
 * @param {number} b - 第二个数
 * @returns {number} 两数之和
 * @throws {Error} 如果参数不是数字，抛出错误
 *
 * @example
 * add(1, 2) // 3
 * add(-1, 1) // 0
 */
function add(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new Error('参数必须是数字')
  }
  return a + b
}

// 解释：JSDoc 格式，提供详细的函数说明
```

**解释**：JSDoc 格式提供详细的函数说明，包括参数、返回值、异常、示例。

---

#### TODO 注释
```javascript
// ✅ 正确：TODO 注释
// TODO: 实现用户权限验证
function deleteUser(id) {
  // ...
}

// ✅ 或使用更详细的格式
// TODO(john): 实现用户权限验证，预计 2024-01-15 完成
function deleteUser(id) {
  // ...
}

// 解释：TODO 注释标记需要完成的任务
```

**解释**：TODO 注释标记需要完成的任务，包括负责人、截止日期等信息。

---

#### FIXME 注释
```javascript
// ✅ 正确：FIXME 注释
// FIXME: 处理边界情况（空数组）
function sumArray(arr) {
  return arr.reduce((sum, num) => sum + num, 0)  // 空数组会返回 0，可能不符合预期
}

// 解释：FIXME 注释标记需要修复的代码
```

**解释**：FIXME 注释标记需要修复的代码，比 TODO 更紧急。

### 3. 代码格式

#### 缩进和空格
```javascript
// ✅ 正确：使用 2 空格缩进
function add(a, b) {
  return a + b
}

// ❌ 错误：使用 Tab 缩进（除非项目约定）
function add(a, b) {
	return a + b  // 不推荐
}

// ✅ 正确：运算符前后添加空格
const sum = a + b
const result = (x * y) + z

// ❌ 错误：运算符前后没有空格
const sum = a+b  // 不推荐
const result = (x*y)+z  // 不推荐
```

**解释**：
- 推荐使用 2 空格缩进
- 运算符前后添加空格，提高可读性

---

#### 分号使用
```javascript
// ✅ 正确：始终使用分号
const name = 'John'
const age = 30

// ✅ 或根据项目约定不使用分号（ASI 自动插入）
const name = 'John'
const age = 30

// ❌ 错误：混用分号
const name = 'John'
const age = 30  // 不一致
```

**解释**：
- 始终使用分号是安全的选择
- 如果项目约定不使用分号，保持一致即可
- 不要混用分号

---

#### 对象和数组格式
```javascript
// ✅ 正确：对象和数组单行
const user = { name: 'John', age: 30 }
const numbers = [1, 2, 3]

// ✅ 正确：对象和数组多行（复杂情况）
const user = {
  name: 'John',
  age: 30,
  address: {
    city: 'NYC',
    country: 'USA'
  }
}

const numbers = [
  1,
  2,
  3,
  4,
  5
]

// ✅ 正确：末尾逗号（减少 diff 冲突）
const user = {
  name: 'John',
  age: 30,  // ✅ 末尾逗号
}

const numbers = [
  1,
  2,
  3,  // ✅ 末尾逗号
]

// 解释：末尾逗号可以减少 git diff 冲突
```

**解释**：
- 简单对象和数组单行
- 复杂对象和数组多行
- 使用末尾逗号，减少 git diff 冲突

#### 单一职责原则
```javascript
// ❌ 错误：函数职责过多
function processUser(user) {
  // 验证用户
  if (!user.name || !user.email) {
    throw new Error('用户信息不完整')
  }

  // 格式化用户
  const formattedUser = {
    name: user.name.trim(),
    email: user.email.toLowerCase()
  }

  // 保存用户
  database.save(formattedUser)

  // 发送邮件
  emailService.send(formattedUser.email)

  return formattedUser
}

// ✅ 正确：拆分为多个函数
function validateUser(user) {
  if (!user.name || !user.email) {
    throw new Error('用户信息不完整')
  }
}

function formatUser(user) {
  return {
    name: user.name.trim(),
    email: user.email.toLowerCase()
  }
}

function saveUser(user) {
  database.save(user)
}

function sendWelcomeEmail(user) {
  emailService.send(user.email)
}

// 组合函数
function processUser(user) {
  validateUser(user)
  const formattedUser = formatUser(user)
  saveUser(formattedUser)
  sendWelcomeEmail(formattedUser)
  return formattedUser
}

// 解释：每个函数只做一件事，提高可读性和可维护性
```

**解释**：每个函数只做一件事，符合单一职责原则，提高可读性和可维护性。

### 2. 组件封装

#### Vue 组件封装
```vue
<!-- UserCard.vue -->
<template>
  <div class="user-card">
```

### 3. 状态管理

#### 状态提升
```javascript
// ❌ 错误：状态分散
function Parent() {

    <>
      <Child1 />
      <Child2 />
    </>
  )
}

function Child1() {
  const [count, setCount] = useState(0)

  // 无法访问 Child1 的 count

  return <button onClick={onIncrement}>点击 {count}</button>
}

function Child2({ count }) {

}

// 解释：状态提升到共同父组件，实现共享
```

**解释**：状态提升到共同父组件，实现共享状态。

### 4. 错误处理

#### 统一错误处理
```javascript
// utils/errorHandler.js
export function handleApiError(error) {
  if (error.response) {
    // 服务器返回错误
    const { status, data } = error.response

    switch (status) {
      case 401:
        // 未授权，跳转到登录页
        window.location.href = '/login'
        break
      case 403:
        // 无权限
        alert('没有权限')
        break
      case 500:
        // 服务器错误
        alert('服务器错误，请稍后重试')
        break
      default:
        alert(data.message || '请求失败')
    }
  } else if (error.request) {
    // 请求未发送
    alert('网络错误，请检查网络连接')
  } else {
    // 其他错误
    alert('未知错误')
  }

  throw error  // 重新抛出，供上层处理
}

// 使用
import { handleApiError } from '@/utils/errorHandler'

async function fetchUser() {
  try {
    const response = await api.getUser()
    return response.data
  } catch (error) {
    handleApiError(error)
  }
}

// 解释：统一错误处理，提高一致性
```

**解释**：统一错误处理，提高一致性和用户体验。

#### 错误边界（React）
```jsx
// ErrorBoundary.jsx
class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  static getDerivedStateFromError(error) {
    return { hasError: true, error }
  }

  componentDidCatch(error, errorInfo) {
    console.error('Error caught by boundary:', error, errorInfo)
    // 可以上报到错误监控服务
  }

  render() {
    if (this.state.hasError) {
      return <div>出错了: {this.state.error.message}</div>
    }

    return this.props.children
  }
}

// 使用
<ErrorBoundary>
  <App />
</ErrorBoundary>
```

#### 性能监控
```javascript
// utils/performance.js
export const performance = {
  // 记录函数执行时间
  measure: (fn, name) => {
    return async (...args) => {
      const start = performance.now()
      try {
        const result = await fn(...args)
        const end = performance.now()
        console.log(`[${name}] 耗时: ${(end - start).toFixed(2)}ms`)
        return result
      } catch (error) {
        const end = performance.now()
        console.error(`[${name}] 耗时: ${(end - start).toFixed(2)}ms, 错误:`, error)
        throw error
      }
    }
  },

  // 记录页面加载时间
  measurePageLoad: () => {
    window.addEventListener('load', () => {
      const timing = performance.timing
      const loadTime = timing.loadEventEnd - timing.navigationStart
      console.log(`页面加载时间: ${loadTime}ms`)
    })
  }
}

// 使用
import { performance } from '@/utils/performance'

const fetchData = performance.measure(
  async () => {
    return await api.fetchData()
  },
  'fetchData'
)

// 解释：性能监控，记录函数执行时间和页面加载时间
```

**解释**：性能监控记录函数执行时间和页面加载时间，便于性能优化。

## 四、常见问题

### 1. 什么是单一职责原则？
每个函数/类只做一件事，提高可读性和可维护性。

### 2. 什么是依赖注入？
将依赖作为参数传入，而不是硬编码，提高可测试性和灵活性。

### 3. ES Module 和 CommonJS 的区别？
- ES Module：import/export，静态分析，支持 Tree Shaking
- CommonJS：require/module.exports，动态加载

### 4. 什么是状态提升？
将状态提升到共同父组件，实现共享状态。

### 5. 错误边界的作用？
捕获子组件错误，防止整个应用崩溃。

### 6. ESLint 和 Prettier 的区别？
- ESLint：检查代码质量和风格
- Prettier：自动格式化代码

### 7. Husky 和 lint-staged 的作用？
- Husky：Git hooks 管理
- lint-staged：只检查和格式化暂存区的文件

### 8. 什么是 Code Review？
代码审查，提高代码质量和团队协作。

### 9. 什么是 CI/CD？
- CI：持续集成，自动测试
- CD：持续部署，自动发布

### 10. 什么是 Git Flow？
Git 分支管理模型，包含 master、develop、feature、release、hotfix 分支。
